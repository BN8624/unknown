## TASK_004 결정

- 공격력·체력·방어력의 비용과 강화 횟수를 각각 독립 관리(`*_upgrade_cost`, `*_upgrade_count`). 한 능력치를 올려도 다른 능력치 첫 비용은 그대로.
- 실제 피해 숫자를 대상 근처에 잠깐 표시. 방어력 강화의 효과(적 피해 4→3)를 화면에서 바로 확인하기 위한 최소 표현으로, 색 등급·치명타·파티클은 넣지 않음.
- 검증은 골드를 무작정 공격력에 쓰지 않고 체력 1회 → 방어력 1회 → 공격력 2회+ 순서로 사용해 세 강화를 모두 검증.

## TASK_005 결정

- 적 종류 추가보다 특성 시스템을 먼저 구현. 강타·연격·반격이 서로 다른 플레이로 느껴지는지를 먼저 검증해야, 그 차이를 드러낼 적(체력형·다수형·강공격형)을 의미 있게 설계할 수 있어서다.
- 우선순위: 기본 공격은 강타 발동 확인 → 아니면 일반 → 일반일 때만 연격 판정. 강타 공격에서 연격을 발동시키지 않은 이유는 한 번의 공격이 강타이면서 동시에 연격이면 효과·연출이 겹쳐 어떤 특성 때문인지 구분이 흐려지기 때문.
- 반격은 적 공격에 대한 별도 반응이라 반격 공격에서 강타 카운트 증가·연격 발동을 막았다. 강타 카운트는 "용병의 기본 공격"만 세야 4타째 강타 타이밍이 일정해진다.
- 일반 플레이는 정본 확률(연격 30%, 반격 25%)을 쓰되, 헤드리스 검증은 확률 결과로 성공/실패가 갈리면 안 되므로 force 플래그로 강제 발동/미발동 경로를 두고 피해 공식·동작 경로만 결정적으로 검증.
- 특성 초기화는 무료. 프로토타입 단계에서 세 특성을 부담 없이 바꿔 비교하는 것이 "서로 다른 플레이인가"를 판단하는 핵심이라 비용·확인 팝업을 두지 않았다.

- (TASK_011 아이폰 피드백) 특성 투자 후 "닫기"가 바로 안 눌리고 여러 번 눌러야 닫히는 문제. 1차로 버튼 비활성화 타이밍(같은 터치 중 `disabled=true`)을 의심해 UI 갱신을 `call_deferred`로 미뤘으나, 이어진 제보("오우거 진입 전 특성창 열고 들어가면 처치해야 닫힘")로 진짜 원인이 드러남.
- (진짜 원인) Godot의 Control 입력 픽킹은 z_index가 아니라 **트리 순서**(나중 형제가 위)를 따른다. 특성 패널(트리상 먼저 생성)보다 **전투 중 동적 생성되는 비상호작용 노드**(떠오르는 피해 숫자 Label, 잔상 ColorRect, 적·보스 도형과 체력 바)가 트리상 뒤에 추가돼 입력 우선순위가 더 높았고, 이들이 기본 `mouse_filter=STOP`이라 패널의 닫기 버튼(y560 부근, 전투 영역과 겹침) 위에서 터치를 가로챘다. 오우거는 전투가 길어 피해 숫자가 계속 떠 항상 막혔고, 처치하면 전투가 멈춰 그제야 닫혔다. 같은 이유로 보스 정보 패널(전체 화면 Control)도 강화 버튼을 막았다.
- (해결) 상호작용하지 않는 모든 표시를 `MOUSE_FILTER_IGNORE`로: 피해 숫자·잔상, 적·보스·용병 도형과 체력 바(`_ignore_enemy_mouse`), 상태·경험치·알림·준비·진행 라벨과 배경 줄무늬(`_ignore_decorative_mouse`), 보스 정보·결과 패널(`_ignore_mouse`). 버튼과 특성 패널 배경만 STOP으로 남겨 입력을 받게 했다. 원칙: 화면의 표시 노드는 기본적으로 입력을 통과시키고, 실제 버튼/모달만 입력을 잡는다.
