## TASK_001 결정

- 엔진: 설치된 Godot 4.7.stable 사용 (`C:\Users\USER\godot-engine\Godot_v4.7-stable_win64.exe`).
- 임시 도형은 별도 에셋 없이 코드에서 `ColorRect`로 생성. 용병/적/체력바/배경 줄무늬 모두 동일 방식.
- 상태 머신은 범용 프레임워크 없이 `enum { WALK, COMBAT, DEAD }`와 변수만으로 최소 처리.
- 유닛은 별도 클래스/씬으로 분리하지 않고 `Dictionary`(merc/enemy)로 보관. 단일 화면·단일 적 종류라 과분리 회피.
- 피해 계산은 `피해량 = 공격력`으로 단순화하고 무작위를 쓰지 않아 같은 조건에서 같은 결과가 나오게 함. (방어력·치명타·확률 효과는 TASK_001 범위 밖)
- 임시 수치는 TASK_001 카드 값(용병 100/10/1.0초, 적 30/4/1.5초)을 사용. `UNKNOWN.md` 15·16장의 정식 임시 수치(용병 120/12, 늑대 32/5 등)와 다른데, 이는 카드가 명시한 첫 루프 검증용 값이라 카드를 우선함. 성장 시스템 붙일 때 `UNKNOWN.md` 값으로 맞춘다.
- 타격 표현은 카드가 요구한 "2가지 이상" 중 3가지 사용: 공격 시 전진 모션, 피격 시 밝아짐, 체력 바 감소.
- 검증: 헤드리스 실행이 불가능하면 자동 검증을 못 하므로, `-- --verify` 인자를 받으면 `Engine.time_scale=8`로 가속하고 적 6마리 처치 후 자동 종료하며 로그를 남기도록 함. 수치는 그대로, 시간만 가속이라 결과는 동일.

## TASK_002 결정

- 골드는 적 처치당 +5(카드 값). 강화 항목은 공격력 하나만. 체력·방어력·경험치·레벨은 범위 밖이라 미구현.
- 강화 비용은 직전 × 1.4를 정수로 반올림: 20 → 28 → 39 → 55. 골드가 정수라 반올림으로 표시·차감을 일치시킴.
- "처치 속도 단축"은 처치당 타격 수로 검증. 적 체력 30 기준 공격력 10·12·14는 3타, 16부터 2타라 강화 효과가 눈에 보임.
- 검증(--verify): 처치마다 가능한 만큼 자동 강화하고, 첫 처치 타격 수보다 적은 타격으로 잡히면 PASS. 헤드리스에서 kill=19, 공격력 16에서 3타→2타 확인.

## TASK_003 결정

- 경험치/레벨 수치는 카드와 `UNKNOWN.md` 규칙을 그대로 사용(처치당 5, 필요 = 20 + 레벨 × 5, 레벨업 공격력 +1·최대 체력 +5·완전 회복).
- 골드 강화 공격력과 레벨업 공격력을 구분하기 위해 `attack_upgrade_count`를 추적. 검증은 `merc.atk == 시작 + 강화×2 + 레벨업×1` 불변식으로 매 처치 확인해 둘을 혼동하지 않게 함.
- 검증 종료 조건을 바꿈: TASK_002(골드 강화→더 적은 타격)와 TASK_003(첫 레벨업 상태)이 모두 통과해야 종료 코드 0. 하나만 통과하면 종료하지 않고, 실패 시 어긋난 값을 로그로 남기고 코드 1.
- UI 배치: 상단 상태를 두 줄로 나누고(레벨·EXP·골드·처치 / 용병 HP·적 HP), 경험치 바는 그 아래(체력 바와 색을 보라로 구분), 레벨업 문구는 화면 중앙에 약 1초만 표시.

## TASK_004 결정

- 방어력(용병 2) 도입에 맞춰 일반 적 공격력을 4 → 6으로 올림. 6 - 2 = 4로 기본 실제 피해를 그대로 둬 기존 전투 속도·생존 시간을 보존하기 위해서다. 난이도 상승이 아니라 시스템 도입에 따른 상쇄 조정.
- 체력 강화는 완전 회복이 아니라 최대·현재 체력을 함께 +20. 잃은 체력의 양을 유지해 "회복 버튼"처럼 느껴지지 않게 하고, 레벨업의 완전 회복과 역할을 구분하기 위해서다.
- 공격력·체력·방어력의 비용과 강화 횟수를 각각 독립 관리(`*_upgrade_cost`, `*_upgrade_count`). 한 능력치를 올려도 다른 능력치 첫 비용은 그대로.
- 실제 피해 숫자를 대상 근처에 잠깐 표시. 방어력 강화의 효과(적 피해 4→3)를 화면에서 바로 확인하기 위한 최소 표현으로, 색 등급·치명타·파티클은 넣지 않음.
- 검증은 골드를 무작정 공격력에 쓰지 않고 체력 1회 → 방어력 1회 → 공격력 2회+ 순서로 사용해 세 강화를 모두 검증.

## TASK_005 결정

- 적 종류 추가보다 특성 시스템을 먼저 구현. 강타·연격·반격이 서로 다른 플레이로 느껴지는지를 먼저 검증해야, 그 차이를 드러낼 적(체력형·다수형·강공격형)을 의미 있게 설계할 수 있어서다.
- 우선순위: 기본 공격은 강타 발동 확인 → 아니면 일반 → 일반일 때만 연격 판정. 강타 공격에서 연격을 발동시키지 않은 이유는 한 번의 공격이 강타이면서 동시에 연격이면 효과·연출이 겹쳐 어떤 특성 때문인지 구분이 흐려지기 때문.
- 반격은 적 공격에 대한 별도 반응이라 반격 공격에서 강타 카운트 증가·연격 발동을 막았다. 강타 카운트는 "용병의 기본 공격"만 세야 4타째 강타 타이밍이 일정해진다.
- 일반 플레이는 정본 확률(연격 30%, 반격 25%)을 쓰되, 헤드리스 검증은 확률 결과로 성공/실패가 갈리면 안 되므로 force 플래그로 강제 발동/미발동 경로를 두고 피해 공식·동작 경로만 결정적으로 검증.
- 특성 초기화는 무료. 프로토타입 단계에서 세 특성을 부담 없이 바꿔 비교하는 것이 "서로 다른 플레이인가"를 판단하는 핵심이라 비용·확인 팝업을 두지 않았다.

## 보류 결정 — 특성 위력 증가는 2·3단계로

- (2026-06-24, 아이폰 플레이) 템포 단축(공격 0.7/1.0초 등) 후 강타·연격·반격 차이는 확실히 보임. 다만 1단계 위력이 "살짝 약함". 이는 의도된 절제이며, 위력 증가는 특성 2·3단계의 몫으로 둔다(UNKNOWN.md 설계). 1단계 배율은 정본 값이라 임의로 올리지 않고, 2·3단계는 현재 프로토타입 범위 밖이라 지금 구현하지 않는다.
- 사용자 확인: 현재 템포가 좋음(이전이 느렸음). 템포 수치 유지. 전투 간격을 0.7/1.0초로 줄이고 적 접근·재등장을 빠르게 해 특성 효과가 자주 보이게 함.
- 연격은 청록 잔상·발광, 반격은 방어(파란 번쩍)→되받음(주황 잔상)으로 연출을 강조. 강타는 기존 유지.
