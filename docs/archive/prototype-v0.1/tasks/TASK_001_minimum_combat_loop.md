# TASK 001 — 최소 자동전투 루프 구현

## 작업 목적

`unknown`의 첫 번째 플레이 가능한 화면을 만든다.

이번 작업의 목표는 다음 전투 흐름을 눈으로 확인하는 것이다.

```text
용병 자동 전진
→ 적 등장
→ 전진 정지
→ 서로 자동 공격
→ 적 처치
→ 용병 다시 전진
→ 다음 적 등장
```

완성 게임을 만드는 작업이 아니다.

골드, 경험치, 강화, 특성 등은 아직 구현하지 않는다.

---

## 먼저 읽을 문서

작업 전에 다음 순서로 읽는다.

1. `CLAUDE.md`
2. `UNKNOWN.md`
3. `HANDOFF.md`
4. `checklist.md`
5. `context-notes.md`

`UNKNOWN.md`는 게임 기획 정본이다.

이번 작업에서 `UNKNOWN.md`와 `CLAUDE.md`는 수정하지 않는다.

---

## 기술 기준

이번 프로토타입은 Godot 4로 구현한다.

저장소에 Godot 프로젝트가 없다면 최소 프로젝트를 새로 생성한다.

기준은 다음과 같다.

* 엔진: Godot 4.
* 게임 형태: 2D.
* 화면 방향: 세로형.
* 기준 해상도: 540 × 960.
* 실행 환경: 우선 PC 로컬 실행.
* 외부 에셋과 외부 플러그인은 사용하지 않는다.
* 네트워크와 서버는 사용하지 않는다.

Godot 버전의 세부 차이로 동작하지 않는 기능은 현재 설치된 Godot 4 환경에 맞춰 단순하게 수정한다.

---

## 구현 범위

### 1. 메인 전투 화면

단일 전투 화면을 만든다.

화면은 크게 다음 영역으로 구성한다.

```text
상단:
프로토타입 상태 표시.

중앙:
용병, 적, 배경이 표시되는 전투 영역.

하단:
이번 작업에서는 비워두거나 간단한 안내 문구만 표시.
```

완성형 UI는 만들지 않는다.

---

### 2. 임시 용병

용병은 임시 도형으로 표현한다.

* 화면 왼쪽 약 30% 지점에 위치한다.
* 이동 중에도 화면상의 위치는 거의 고정한다.
* 배경이 왼쪽으로 움직이며 전진하는 것처럼 표현한다.
* 적이 공격 거리 안에 들어오면 전진을 멈춘다.
* 적을 처치하면 다시 전진한다.

필요한 상태는 다음과 같다.

```text
WALK
COMBAT
HIT
DEAD
```

복잡한 범용 상태머신 프레임워크는 만들지 않는다.

현재 기능에 필요한 최소 상태 처리만 구현한다.

---

### 3. 임시 적 한 종류

적은 임시 도형으로 표현한다.

* 화면 오른쪽에서 등장한다.
* 용병에게 접근한다.
* 전투 위치에 도달하면 멈춘다.
* 일정 간격으로 용병을 공격한다.
* 체력이 0이 되면 사라진다.
* 짧은 대기 후 다음 적이 다시 등장한다.

이번 작업에서는 적 종류를 하나만 만든다.

늑대, 고블린, 방패병을 따로 구현하지 않는다.

---

### 4. 기본 자동 공격

용병과 적은 플레이어 입력 없이 자동으로 공격한다.

임시 능력치는 다음 값을 사용한다.

#### 용병

```text
최대 체력: 100
공격력: 10
공격 간격: 1.0초
```

#### 적

```text
최대 체력: 30
공격력: 4
공격 간격: 1.5초
```

이번 작업의 피해 계산은 단순하게 처리한다.

```text
피해량 = 공격력
```

아직 구현하지 않는 항목은 다음과 같다.

* 방어력.
* 치명타.
* 회피.
* 속성.
* 확률 효과.
* 강타.
* 연격.
* 반격.

`Math.random` 또는 무작위 전투 결과를 사용하지 않는다.

같은 조건에서는 같은 결과가 나와야 한다.

---

### 5. 공격 표현

완성 애니메이션은 필요 없지만, 공격 여부는 분명히 보여야 한다.

최소한 다음 중 두 가지 이상을 사용한다.

* 공격 순간 용병이나 적이 앞으로 짧게 움직인다.
* 피격 대상이 짧게 흔들린다.
* 피격 대상이 잠시 밝아지거나 깜박인다.
* 간단한 타격 표시가 나타난다.
* 체력 바가 감소한다.

강한 파티클이나 복잡한 애니메이션 시스템은 만들지 않는다.

이번 단계의 목적은 임시 도형만으로도 전투 흐름이 읽히는지 확인하는 것이다.

---

### 6. 체력 표시

용병과 적의 현재 체력을 확인할 수 있어야 한다.

* 용병 체력 바.
* 적 체력 바.
* 공격을 받으면 즉시 감소.
* 적이 교체되면 적 체력 바 초기화.

숫자와 체력 바 중 하나만 사용해도 되지만, 전투 상태를 쉽게 확인할 수 있어야 한다.

---

### 7. 적 처치 후 반복

적 체력이 0이 되면 다음 순서로 진행한다.

```text
적 사망 표현
→ 적 제거
→ 약 0.5~1초 대기
→ 용병 전진 재개
→ 새 적 등장
→ 다시 전투
```

사용자 입력 없이 이 루프가 계속 반복되어야 한다.

적을 처치한 뒤 전투가 멈추거나 새 적이 겹쳐서 등장하면 안 된다.

---

### 8. 용병 사망 처리

현재 수치에서는 보통 용병이 적보다 먼저 죽지 않아야 한다.

다만 예외 상황에서도 게임이 정지하지 않도록 최소 처리를 추가한다.

용병이 죽으면 다음처럼 처리한다.

```text
용병 사망 표시
→ 약 1초 대기
→ 용병 체력 전부 회복
→ 적 상태 초기화
→ 전투 루프 재시작
```

사망 패널티나 재화 손실은 구현하지 않는다.

---

## 코드와 구조 원칙

* 현재 작업에 필요한 최소한의 씬과 스크립트만 만든다.
* 미래 시스템을 위한 범용 전투 엔진을 만들지 않는다.
* 컴포넌트 구조를 과도하게 분리하지 않는다.
* 밸런스 수치는 한 곳에서 쉽게 수정할 수 있게 둔다.
* 의미를 알기 어려운 숫자를 여러 파일에 흩어놓지 않는다.
* 외부 패키지와 애드온을 추가하지 않는다.
* 임시 도형은 Godot 기본 노드나 코드로 생성해도 된다.
* 사용하지 않는 빈 폴더와 미래용 파일을 미리 만들지 않는다.

프로젝트가 이미 존재한다면 기존 구조를 먼저 확인하고 최소 변경으로 구현한다.

이번 작업에서 다음 기능을 추가하지 않는다.

* 골드.
* 경험치.
* 레벨업.
* 공격력 강화.
* 체력 강화.
* 방어력 강화.
* 강타.
* 연격.
* 반격.
* 특성 포인트.
* 프리셋.
* 엘리트.
* 보스.
* 빚.
* 명성.
* 장비.
* 저장과 불러오기.
* 오프라인 보상.
* 사운드 시스템.
* 완성형 아트.
* 모바일 빌드와 배포.
* 설정 화면.
* 범용 게임 제작 프레임워크.
* 골렘 코드 재사용.

작업 중 좋은 아이디어가 떠나더라도 이번 구현에는 넣지 않는다.

필요하다면 `context-notes.md`의 보류 항목에 한 줄로만 기록한다.

## 완료 조건

다음 조건을 모두 만족해야 이번 작업이 완료된다.

* Godot에서 프로젝트가 오류 없이 실행된다.
* 화면이 세로 비율로 표시된다.
* 용병이 전진하는 것처럼 보인다.
* 적이 오른쪽에서 등장한다.
* 적이 접근하면 전진이 멈춘다.
* 용병과 적이 자동으로 공격을 주고받는다.
* 공격과 피격이 화면에서 구분된다.
* 체력이 실제로 감소한다.
* 적이 죽으면 제거된다.
* 적 처치 후 용병이 다시 전진한다.
* 다음 적이 등장한다.
* 사용자 입력 없이 전투 루프가 반복된다.
* 최소 5마리 이상 연속 처치해도 흐름이 깨지지 않는다.
* 실행 중 콘솔에 반복 오류가 발생하지 않는다.
* 범위 밖의 성장 시스템이 추가되지 않았다.

## 검증 방법

구현 후 다음을 직접 확인한다.

1. 프로젝트를 실행한다.
2. 최소 60초 동안 별도 입력 없이 지켜본다.
3. 적이 최소 5회 등장하고 처치되는지 확인한다.
4. 용병의 전진과 전투 전환이 자연스럽게 이어지는지 확인한다.
5. 공격 순간과 피격 순간을 눈으로 구분할 수 있는지 확인한다.
6. 적이 중복 생성되거나 전투가 멈추지 않는지 확인한다.
7. Godot 출력 창과 디버거에 반복 오류가 없는지 확인한다.

실제로 실행하지 못한 항목은 검증 완료로 표시하지 않는다.
