# CLAUDE.md — unknown 작업 규칙

## 프로젝트 목적

`unknown`은 캐릭터 한 명을 성장시키는 세로형 방치형 RPG 프로젝트다.

이 프로젝트의 현재 목표는 완성 게임 제작이 아니다.

> 15분 동안 플레이할 수 있는 프로토타입을 만들고, 자동 전투와 성장 선택이 실제로 재미있는지 검증한다.

* 골렘과 별개의 신규 프로젝트다.
* 골렘과 포지의 코드 및 구조를 가져오지 않는다.
* 멀티에이전트 자동 제작 시스템을 만들지 않는다.
* 재미가 검증되기 전에는 콘텐츠와 장기 시스템을 확장하지 않는다.

## 작업 흐름 (TASK 013부터)

TASK 013부터는 매번 새 Markdown 작업지시서 파일을 만들지 않는다.

* **작업의 정본은 GitHub Issue다.** Issue 하나 = 작업 주제 하나. 본문에 작업지시서, 댓글에 Claude 결과·사용자 판단·수정 요청을 남기고, 완료되면 Issue를 닫는다.
* **로컬에는 `CURRENT_TASK.md` 하나만** 유지한다. 지금 실행할 작업 하나만 담고, 새 작업을 시작하면 기존 내용을 지우고 덮어쓴다. 맨 위에는 Issue 번호·상태·정본 표시만 둔다.
* Issue 확인은 가능하면 `gh issue view <번호> --comments`로 읽는다. GitHub CLI 접근이 불안정하면 사용자가 Issue 본문을 `CURRENT_TASK.md`에 붙여 둔 내용을 본다.
* **Issue 본문/댓글과 `CURRENT_TASK.md` 내용이 다르면 GitHub Issue를 우선한다.**
* `docs/archive/`의 과거 TASK 파일은 보관용 이력이다. **현재 지시로 사용하지 않는다.**
* **GPT에게 Issue를 줄 때는 항상 아래 평문 JSON/raw URL을 함께 준다**(GPT는 GitHub HTML을 자주 못 읽는다). `<N>`=이슈 번호, `<CID>`=댓글 ID, `<경로>`=레포 루트 기준 파일 경로.
  * 본문: `https://api.github.com/repos/BN8624/unknown/issues/<N>`
  * 댓글 전체: `https://api.github.com/repos/BN8624/unknown/issues/<N>/comments`
  * **특정 댓글(가장 확실)**: `https://api.github.com/repos/BN8624/unknown/issues/comments/<CID>` — `<CID>`는 댓글 URL `...#issuecomment-<CID>` 끝 숫자, 또는 `gh api repos/BN8624/unknown/issues/<N>/comments --jq '.[].id'`로 확인.
  * 파일 원문(raw): `https://raw.githubusercontent.com/BN8624/unknown/main/<경로>` (대소문자 구분 — 예: `ART_STYLE_GUIDE.md`). 푸시 직후면 1분 내 반영.
  * JSON `body` 필드가 본문/댓글 평문이다. GPT가 "댓글이 N개뿐"이라고 하면 **이슈 번호가 맞는지부터 확인**(닫힌 이슈의 옛 번호를 보는 경우가 많음).

## 문서 우선순위

문서의 역할을 다음과 같이 고정한다(중복 금지).

1. 지금 실행할 작업 하나는 `CURRENT_TASK.md`에 담는다. 정본은 그 위에 적힌 GitHub Issue다.
2. 현재 위치·완료 지점·남은 문제·다음 한 단계는 `HANDOFF.md`만 본다(과거 TASK 전체 내역을 반복하지 않는다).
3. 게임 기획과 확정 규칙·핵심 수치의 정본은 `UNKNOWN.md`다(작업 과정·시행착오는 넣지 않는다).
4. 작업 규칙의 정본은 `CLAUDE.md`다.
5. 결정 이유(무엇을 했는지가 아니라 왜 그렇게 했는지)만 `context-notes.md`에 기록한다.
6. 큰 단계만 `checklist.md`에 기록한다(TASK별 세부 구현 항목을 계속 쌓지 않는다).
7. 작업별 전체 대화·실행 기록은 GitHub Issue에 남긴다.

문서가 충돌할 경우 우선순위는 다음과 같다.

```text
사용자의 최신 명시 지시
→ GitHub Issue 본문·댓글 (현재 작업의 정본)
→ CURRENT_TASK.md의 현재 작업
→ HANDOFF.md의 현재 상태
→ UNKNOWN.md의 확정 기획
→ CLAUDE.md의 작업 규칙
→ context-notes.md
→ checklist.md
```

`UNKNOWN.md`의 기획을 Claude가 임의로 변경하지 않는다.

구현 과정에서 기획 변경이 필요해 보이면 코드로 우회하지 말고, 변경 필요성과 이유를 사용자에게 보고한다.

## 역할 구분

### 사용자

* 게임의 최종 결정권자다.
* 직접 플레이하고 재미 여부를 판단한다.
* 기획 갈림길에서 방향을 선택한다.
* 기능 추가와 범위 확장을 승인한다.

### GPT

* 기획을 구체적인 규칙으로 정리한다.
* 구현 범위를 제한한다.
* Claude에게 전달할 작업 단위를 작성한다.
* 결과물을 기획과 재미 관점에서 검토한다.

### Claude Code

* 확정된 기획을 실행 가능한 형태로 구현한다.
* 한 번에 확인 가능한 작은 단위로 작업한다.
* 구현 결과와 검증 결과를 짧게 보고한다.
* 사용자를 대신해 재미를 확정하거나 새 기획을 추가하지 않는다.

## 사용자 방식

* 사용자는 코드를 직접 판단하지 않는 바이브코더다.
* 코드 설명보다 무엇이 작동하는지, 무엇이 아직 안 되는지를 사람 말로 설명한다.
* 결정할 사항이 없다면 사소한 단계마다 멈추지 않고 계속 진행한다.
* 실제 기획 갈림길, 범위 확대, 기술 스택 변경이 필요한 경우에만 멈춘다.
* 구현 중에는 장황한 코드 설명을 피한다.
* 결과와 실행 방법, 검증 결과, 남은 위험을 짧게 보고한다.
* 한 번에 복사해 실행할 수 있는 명령을 우선한다.

## 개발 원칙

### 작은 프로토타입 우선

전체 게임을 한 번에 구현하지 않는다.

기본 순서는 다음과 같다.

```text
작은 기능 하나 구현
→ 직접 실행
→ 작동 확인
→ 플레이 감각 확인
→ 다음 기능 추가
```

첫 구현은 가능한 한 다음처럼 작게 유지한다.

> 임시 용병이 오른쪽에서 등장한 임시 적을 자동으로 공격하고, 적을 처치한 뒤 다시 전진하는 단일 전투 루프.

이 전투 루프가 제대로 보이고 작동한 뒤에만 골드, 경험치, 강화, 특성을 추가한다.

### 재미 검증 우선

테스트 통과나 기능 완성만으로 성공이라고 판단하지 않는다.

다음 질문이 더 중요하다.

* 자동 전투를 계속 보고 싶은가.
* 성장 전후의 차이가 보이는가.
* 공격이 맞는 느낌이 있는가.
* 막힌 적을 다시 잡고 싶은가.
* 다음 성장을 기대하게 되는가.

재미가 확인되지 않았다면 다음 지역이나 새로운 시스템을 추가하지 않는다.

### 임시 구현 허용

프로토타입 단계에서는 다음을 허용한다.

* 임시 스프라이트
* 단순 도형
* 임시 배경
* 최소 효과음
* 단순한 UI
* 조정하기 쉬운 임시 수치

다만 임시 구현이어도 전투 흐름과 특성 차이는 확인할 수 있어야 한다.

### 과잉 설계 금지

현재 작업에 필요하지 않은 일반화와 기반 시스템을 만들지 않는다.

다음을 선행 구현하지 않는다.

* 범용 게임 제작 프레임워크
* 플러그인 구조
* 멀티에이전트 시스템
* 복잡한 데이터 파이프라인
* 과도한 설정 파일
* 미래 콘텐츠를 위한 추상화
* 대규모 에디터 도구
* 범용 전투 엔진

두 번째 실제 사용 사례가 생기기 전에는 범용화하지 않는다.

## 기술 선택

기술 스택과 게임 엔진은 사용자가 확정한 내용을 따른다.

아직 확정되지 않은 기술을 Claude가 임의로 정본화하지 않는다.

기술 선택 시 다음을 우선한다.

* 사용자가 쉽게 실행할 수 있는가.
* 모바일 세로 화면을 만들기 적합한가.
* 프로토타입 수정 속도가 빠른가.
* Claude Code가 안정적으로 수정하고 검증할 수 있는가.
* 불필요한 서버나 외부 서비스 없이 로컬에서 실행 가능한가.

기술 스택이 확정되면 `HANDOFF.md`와 `context-notes.md`에 기록한다.

## 구현 규칙

### 한 작업의 크기

한 번의 작업은 사용자가 실행해서 눈으로 확인할 수 있는 단위로 제한한다.

좋은 작업 예시:

* 용병과 적 한 종류의 자동 전투 구현.
* 적 처치 후 다음 적 등장 구현.
* 골드 획득과 공격력 강화 버튼 구현.
* 레벨업과 경험치 표시 구현.
* 강타 1단계만 구현.
* 보스 패배 후 일반 사냥 복귀 구현.

나쁜 작업 예시:

* 게임 전체를 완성한다.
* 모든 성장 시스템을 구현한다.
* 전체 전투 엔진을 범용적으로 설계한다.
* 장비·전직·환생까지 한 번에 추가한다.

### 데이터와 수치

* 프로토타입 수치는 `UNKNOWN.md`의 임시 값을 기준으로 한다.
* 밸런스 수치는 코드 여러 곳에 흩어놓지 않는다.
* 사용자가 쉽게 조정할 수 있는 한 곳에 모은다.
* 수치를 임의로 변경했다면 변경 이유와 영향을 보고한다.
* 재미 검증 전에는 복잡한 자동 밸런싱 시스템을 만들지 않는다.

### 저장

첫 프로토타입의 저장 범위는 다음으로 제한한다.

* 레벨
* 경험치
* 골드
* 기본 강화 횟수
* 선택한 특성
* 보스 처치 여부

저장 데이터 마이그레이션이나 계정 동기화는 만들지 않는다.

### 외부 의존성

* 꼭 필요하지 않은 라이브러리와 플러그인을 추가하지 않는다.
* 외부 서비스, 로그인, 서버, 네트워크 연결을 만들지 않는다.
* 패키지를 추가할 경우 목적과 필요성을 보고한다.
* 같은 기능을 기본 기능으로 단순하게 구현할 수 있다면 기본 기능을 우선한다.

## 검증 규칙

각 작업 후 가능한 범위에서 다음을 확인한다.

* 프로젝트가 오류 없이 실행되는가.
* 새 기능이 실제 화면에서 작동하는가.
* 기존 전투 흐름이 깨지지 않았는가.
* 저장과 불러오기가 필요한 작업이라면 정상 동작하는가.
* 모바일 세로 화면에서 UI가 겹치지 않는가.
* 콘솔에 반복 오류가 발생하지 않는가.

자동 테스트가 유용한 계산과 상태 로직에는 테스트를 작성할 수 있다.

다만 애니메이션, 타격감, 성장 체감과 같은 재미 요소를 테스트 통과로 대신 판단하지 않는다.

재미 판단은 사용자 플레이 결과를 따른다.

---

## 작업 중 변경 제한

다음 변경은 사용자 승인 없이 하지 않는다.

* 게임 장르 변경
* 핵심 루프 변경
* 화면 방향 변경
* 캐릭터 수 증가
* 엔진 또는 주요 기술 스택 변경
* `UNKNOWN.md`에 없는 대형 시스템 추가
* 프로토타입에서 완성 게임 제작으로 범위 확대
* 골렘이나 포지의 코드 도입
* 프로젝트 구조 전면 재작성

버그 수정이나 현재 작업에 필요한 작은 내부 정리는 별도 승인 없이 할 수 있다.

단, 대규모 리팩터링은 먼저 필요성을 보고한다.

## 문서 관리

### HANDOFF.md

현재 상태만 기록한다.

반드시 다음 항목을 유지한다.

```markdown
## 현재 목표

## 지금까지 완료

## 현재 작업

## 다음 액션

## 실행 방법

## 검증 상태

## 남은 위험
```

오래된 작업 이력과 긴 결정 이유를 누적하지 않는다.

### context-notes.md

다음 내용을 기록한다.

* 중요한 결정의 이유
* 검토했지만 채택하지 않은 안
* 재미 검증에서 발견된 문제
* 나중에 검토할 보류 아이디어
* 임시 수치를 바꾼 이유

### checklist.md

완료 여부만 관리한다.

* 완료한 항목은 체크한다.
* 세부 토론과 긴 설명을 적지 않는다.
* `UNKNOWN.md`의 프로토타입 범위를 기준으로 구성한다.

### UNKNOWN.md

게임 기획 정본이다.

* Claude가 임의로 수정하지 않는다.
* 구현 결과와 기획이 달라져야 할 경우 먼저 사용자에게 보고한다.
* 사용자 승인 후에만 변경한다.

## 협업 문서 처리

사용자가 특정 Markdown 파일이나 GitHub Issue 내용을 가리키면 해당 내용을 먼저 읽는다.

응답 작성 시 다음을 구분한다.

* `[확인]`: 실제 코드나 문서에서 확인한 내용.
* `[결과]`: 구현하거나 실행해 확인한 내용.
* `[의견]`: Claude의 판단이나 제안.
* `[추측]`: 아직 실행 또는 확인하지 못한 내용.

사용자가 코드 수정 허용 없이 검토만 요청했다면 코드를 수정하지 않는다.

기존 문서의 사용자 작성 부분을 임의로 삭제하거나 바꾸지 않는다.

---

## 보고 형식

보고는 다음 네 항목을 중심으로 짧게 작성한다.

```markdown
## 변경

무엇을 구현하거나 수정했는지 설명한다.

## 검증

무엇을 실행했고 어떤 결과가 나왔는지 설명한다.

## 실행

사용자가 직접 확인할 수 있는 실행 방법을 적는다.

## 남은 위험

아직 확인하지 못했거나 다음 플레이에서 판단해야 할 내용을 적는다.
```

코드의 내부 동작을 장황하게 설명하지 않는다.

사용자가 결정해야 할 사항이 없다면 다음 작업을 이어서 진행할 수 있도록 `HANDOFF.md`의 다음 액션을 명확히 남긴다.

---

## 세션 종료 규칙

세션을 마칠 때 반드시 다음을 확인한다.

1. 실제 변경 내용을 검증한다.
2. `checklist.md`의 완료 항목을 갱신한다.
3. 중요한 결정이 있다면 `context-notes.md`에 기록한다.
4. `HANDOFF.md`의 현재 위치와 다음 액션을 갱신한다.
5. 사용자에게 실행 방법과 남은 위험을 짧게 보고한다.

미완성 기능을 완성된 것처럼 보고하지 않는다.

직접 실행하지 못한 내용은 검증하지 못했다고 명시한다.

---

## 현재 첫 작업 원칙

전체 프로토타입을 한 번에 구현하지 않는다.

첫 작업의 목표는 다음으로 제한한다.

> 세로형 화면에서 임시 용병이 자동으로 전진하고, 오른쪽에서 등장한 임시 적 한 마리와 자동으로 기본 공격을 주고받으며, 적을 처치한 뒤 다시 전진하는 전투 루프를 구현한다.

첫 작업에서는 다음을 아직 구현하지 않는다.

* 골드
* 경험치
* 강화
* 특성
* 엘리트
* 보스
* 빚
* 명성
* 저장

먼저 전투를 화면으로 보았을 때 계속 발전시킬 가치가 있는지를 확인한다.
